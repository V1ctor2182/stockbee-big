## 1. 产品概述

Stockbee 是一套面向美股市场的 AI 量化交易系统，融合宏观分析、多因子评分、强化学习组合优化、LLM 新闻分析和人工审批机制，实现智能化的周度投资组合再平衡。

**核心策略**: Long 7 + Short 1，从 U100 精选宇宙中选股，周度调仓

**目标用户**: Victor（投资组合经理）及 2-3 人量化团队

**长远愿景**: 跑通验证后，整理为开源 AI 量化交易框架（模块化、可插拔、可独立使用）

**核心差异化**:

| 维度 | Stockbee | daily_stock_analysis (DSA) | 传统量化 (QuantConnect) |
|-----|---------|---------------------------|----------------------|
| **定位** | 组合管理大脑 | 个股分析助手 | 策略回测+实盘 |
| **核心输出** | 组合再平衡方案 (Long 7+Short 1) | 单股分析报告（买卖建议） | 用户自定义策略 |
| **决策核心** | LLM Agent 编排多信号源做组合决策 | Multi-agent 分析个股 | 规则/代码 |
| **因子评分** | Alpha158 + IC/ICIR + 动态加权 | 无多因子模型 | 用户自建 |
| **RL** | DQN 选股 + PPO 权重优化 | 无 | 用户自建 |
| **知识图谱** | 两层传播引擎（宏观+公司） | 无 | 无 |
| **组合优化** | 仓位约束 + Beta + 行业分散 | 无（单股建议） | 用户自建 |
| **宏观分析** | 19 FRED 指标 + 经济周期 | 无 | 无 |
| **回测** | Walk-forward + P&L 模拟 | 仅方向准确率 | 完整回测 |
| **决策日志** | Decision Loop（护城河） | 无 | 无 |
| **LLM 安全** | 验证层 (Schema+风控+异常) | 完整性检查 | N/A |
| **成本** | 多模型路由 <$15/月 | LiteLLM 路由 | $20/月起 |
| **Web UI** | Phase 2+ | 成熟 React SPA + Electron | 成熟 |
| **通知** | Phase 2+ | 8+ 渠道（微信/飞书/TG） | 有限 |
| **市场** | 美股 | A股/港股/美股 | 全球 |
| **社区** | 零（建设中） | GitHub Trending | 431K 用户 |

**一句话定位区分**:
- **DSA** 告诉你"这只股票值不值得买"
- **Stockbee** 告诉你"你的组合应该买 7 只什么、空 1 只什么、各配多少权重"
- **QuantConnect** 让你自己写策略来决定

**Stockbee 相对 DSA 的核心优势**:
1. **组合级决策** vs 个股分析 — 真正的投资组合管理，不是一只一只分析
2. **量化因子 + RL** — 数据驱动的选股和权重优化，不只是 LLM 主观判断
3. **知识图谱传播** — 事件沿公司关系图传播的信号放大，DSA 完全没有
4. **宏观经济周期** — 19 FRED 指标驱动行业/风格 Tilt
5. **Decision Loop 护城河** — 决策智能持续积累，越用越准

**DSA 相对 Stockbee 的优势**（Stockbee 需要追赶的）:
1. 成熟的 Web UI + 多渠道通知
2. A股/港股支持（Stockbee 仅美股）
3. YAML 策略系统（用户无需写代码）
4. 社区和生态（GitHub Trending，赞助商）
5. 零成本 GitHub Actions 部署

## 3. 功能需求概览

> 各功能的详细分解见 Feature Hierarchy 文档，技术实现见 Tech Design 文档。这里只列产品层面的需求和验收标准。

### 3.1 投资组合管理 (P0, Phase 1)

**需求**：从 U100 中自动选出 Long 7 + Short 1，生成调仓方案，经人工审批后执行交易。

**验收标准**：
- U100 每月自动重构，覆盖 11 大 GICS 行业
- 单股仓位 ≤ 20%，同行业 ≤ 3 只，净 Beta 目标 ~72%
- 交易通过 Alpaca API 执行，幂等（重试不会重复下单）
- 每笔交易记录完整理由（因子贡献、Tilt 偏好、信号来源）

### 3.2 数据架构 (P0, Phase 1)

**需求**：建立完整的数据管道，覆盖股票行情、新闻、宏观、因子、关系图谱、RL 经验库。

**验收标准**：
- 股票数据覆盖 ~4000 只可投资美股，日线 OHLCV，四层漏斗筛选到 U100
- 新闻数据支持 G1/G2/G3 三级处理流水线
- 宏观数据覆盖 19 个 FRED 指标，point-in-time 对齐无未来偏差
- 两层知识图谱：宏观指标传播图（~30 节点）+ 公司关系传播图（U100，~500 边），支持事件传播计算
- 10 个 Provider 抽象接口，支持 backtest/live 无缝切换
- SQLite 使用 WAL 模式（支持多进程并发读写）
- 全部数据 < 2 GB，本地 SSD 即可

### 3.3 模型体系 (P0, Phase 1-2)

**需求**：LLM 多模型路由 + 小模型本地部署 + RL 选股/权重优化。

**验收标准**：
- LLM 月度 API 成本 < $15，覆盖新闻分类、深度分析、SEC 解析、宏观分析
- FinBERT 情绪准确率 > 80%，LightGBM 预测 IC > 0.03
- RL Walk-forward 回测 Sharpe > 1.8（多 seed 验证）
- Phase 2: SAC 替代 PPO，Ensemble 集合策略应对市场制度切换

### 3.4 LLM Agent 决策与安全 (P0, Phase 1)

**需求**：LLM Agent 作为最终决策编排者，整合因子评分、新闻分析、图谱传播等信号，输出交易建议。所有 LLM 输出必须经过验证层。

> CEO Review 发现的 CRITICAL GAP：LLM 直接驱动交易决策缺乏安全护栏

**验收标准**：
- LLM 输出经过 JSON Schema 验证（结构正确）
- 风控约束自动检查（仓位上限、行业集中度、Beta 约束）
- 异常检测：偏离历史分配范围的决策触发告警
- 新闻文本输入前清洗（防 Prompt Injection）
- 周度再平衡必须人工审批，LLM 只做建议
- 所有 LLM 输入/输出记录日志，支持事后审计

### 3.5 宏观分析与知识图谱 (P0, Phase 1-2)

**需求**：将宏观经济信号转化为行业 Tilt 和风格 Tilt，作为 soft bias 叠加在因子分数上。构建两层知识图谱（宏观指标传播图 + 公司关系传播图），实现事件沿图谱传播的信号放大机制。

**验收标准**：
- `score_final = factor_score + 0.1 × propagation_score + 0.2 × tilt_sector + 0.1 × tilt_style`
- 经济周期分类（扩张/过热/滞胀/衰退/复苏）准确率 > 70%
- 央行语调分析覆盖 FOMC 声明
- 宏观传播图：指标异动（Z-score > 1.0）触发传播，3 层衰减，终端映射行业 Tilt
- 公司传播图：G2 新闻分析触发关系传播（供应链同向、竞争反向），输出 propagation_score
- 宏观 Tilt = 0.6 × 规则 Tilt + 0.4 × 传播图 Tilt
- 传播日志可追溯，审批时展示传播路径

### 3.6 因子评分 (P0, Phase 1)

**需求**：多因子评分系统，动态加权，支持 IC/ICIR 滚动评估。

**验收标准**：
- 覆盖价格动量、质量、价值、情绪、宏观 Beta 五大类
- 60 天滚动 IC/ICIR 评估，自动调整因子权重
- 可选 XGBoost 非线性融合

### 3.7 做空策略 (P0, Phase 1)

> CEO Review 发现的架构问题：反向 ETF ≠ 个股做空，风险属性完全不同

**需求**：明确做空执行方式，区分不同方式的风险属性。

**验收标准**：
- 支持三种做空模式：融券直接做空 / 反向 ETF 代理 / 仅做多（无空头）
- 配置文件中明确选择做空模式
- 回测时区分不同做空模式的表现差异

### 3.8 前端展示与运维功能 (P1-P2)

以下功能属于前端展示或运维辅助，不涉及核心交易逻辑，详见 Feature Hierarchy 和 Tech Design：

| 功能 | 核心要求 | 阶段 |
|------|---------|------|
| **实时监控** | G1/G2/G3 新闻漏斗，Tier-1/2/3 分层行动，ATR×1.5 防守平仓 | Phase 2 |
| **预算治理** | 月度 < $200，70%/85%/95% 三级降级 | Phase 1 |

| **仪表板与报告** | 持仓 P&L、板块热力图、因子 IC 趋势、日/周/月报告 | Phase 1-3 |

## 5. 护城河策略与商业路线图

> 来自 Office Hours (2026-03-24) + Codex 独立审查

### 护城河：被动积累的 LLM 决策智能 (Decision Intelligence)

Stockbee 的护城河不是代码（可以 fork），不是数据源（大家都一样），而是 **积累的决策智能语料库** — 每次 LLM Agent 做出决策，系统自动记录完整的 Decision Record：

```
Decision Record Schema:
{
  timestamp, market_regime, signals_snapshot,
  llm_input_prompt, llm_output_raw, llm_output_parsed,
  decision: {action, ticker, size, confidence},
  rationale,
  risk_check_result,
  outcome_1d, outcome_5d, outcome_20d  (延迟填充)
}
```

**数据飞轮**：用户越多 → 决策样本越大 → 系统能分析"哪类决策在哪种市场环境下表现好" → 决策越准 → 更多用户。用户不需要主动贡献，只要使用就自动产生数据。

### 三步商业路线图

| 阶段 | 用户 | 产品形态 | 变现 | 时间 |
|------|------|---------|------|------|
| **Step 1** | Victor 自己 | 完整交易系统 + Decision Loop | 无（自用） | Now → 6m |
| **Step 2** | 量化爱好者/开发者 | 开源框架 | Open Core（免费框架 + 付费云数据汇聚） | 6-12m |
| **Step 3** | 炒股小白 | SaaS 信号订阅服务 | 订阅费 | 12m+ |

**关键决策**：小白 SaaS 延后到 Step 3（Codex 独立审查认为小白用户高流失、低信任、合规敏感，不适合作为早期客户）。

### Decision Loop 是核心架构

Decision Loop 不是"日志功能"，是护城河的种子。从 Phase 1 第一天开始，每个 LLM 决策都必须记录完整的 Decision Record，存入 SQLite decision_log 表。

## 6. 阶段规划

### Phase 1: MVP (3-6 个月)
投资组合管理 + 因子评分 + 宏观分析 + 基础 RL (DQN+PPO) + LLM Agent 决策层（含验证层）+ **Decision Loop（核心架构）** + 审批工作流 + 预算治理 + 市场日历 + 基础仪表板 + 知识图谱静态图（可查询，辅助审批）

**成功标准**: 系统稳定运行 4 周，月度成本 < $200，所有 LLM 输出经过验证层，**决策日志完整可回放审计，积累 1000+ Decision Records**

### Phase 2: 增强 (3-4 个月)
实时监控 + G1/G2/G3 新闻漏斗 + Tier-1/2/3 分级操作 + RL 升级 (SAC+Ensemble) + 记忆库 + Polymarket 集成 + 知识图谱传播引擎上线（自动传播 + propagation_score 因子整合）+ **决策智能分析**（基于积累的 Decision Records 分析决策模式）

**成功标准**: 实时操作命中率 > 60%，告警响应 < 5 分钟

### Phase 3: 生产优化 + 开源 (2-3 个月)
完整仪表板 + 自动化报告 + 模型性能优化 + 多数据源容错 + **开源整理**（模块化重构、Strategy 插件接口、CLI 工具链、开发者文档、示例策略）

**成功标准**: 99.5% 可用性，GitHub 可公开，开发者可跑通示例

### Phase 4: SaaS 信号服务 (待定)
基于 Step 2 积累的决策智能 + 用户基础，推出面向小白的 SaaS 信号订阅服务。需要验证小白用户留存率和付费意愿后再决定。

## 6. 风险与缓解

| 风险 | 影响 | 缓解方案 |
|------|------|---------|
| LLM Prompt Injection | 错误交易决策 | 验证层 + 新闻文本清洗 + 风控硬约束 + 人工审批 |
| LLM Hallucination | 无效交易指令 | JSON Schema 验证 + 异常检测 |
| 模型过拟合 | 实盘亏损 | Walk-forward 验证、多 seed 训练、等权重回退兜底 |
| API 成本超支 | 运营不可持续 | 预算治理四级降级、廉价模型路由、缓存 |
| 因子/模型失效 | 亏损周期 | 动态因子权重、制度切换集合策略、定期重训 |
| 数据质量 | 决策错误 | 多数据源 fallback、日度卫生检查、异常值过滤 |
| 交易执行延迟 | 滑点成本 | 盘前预生成订单、分批成交 |
| 做空执行 | 风险属性不匹配 | 三种做空模式可配置，回测区分表现 |
| SQLite 并发 | 数据写入冲突 | WAL 模式 |
