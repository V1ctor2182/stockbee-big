## 1. 周度再平衡流程 (Friday Close → Monday Open)

```
周五收盘
  │
  ├─→ 数据更新模块
  │   ├─ 日线 OHLCV (Alpaca Historical)
  │   ├─ 新闻摄入 (NewsAPI)
  │   └─ 宏观指标 (FRED latest)
  │
  ├─→ 因子计算模块
  │   ├─ 技术因子 (表达式引擎)
  │   ├─ 情绪因子 (FinBERT)
  │   └─ 宏观因子 (FRED 组合)
  │
  ├─→ RL 推理模块
  │   ├─ DQN 推理 (100→8 选股)
  │   └─ PPO/SAC 推理 (权重分配)
  │
  ├─→ 宏观倾斜覆盖层 (Soft Overlay)
  │   └─ 根据宏观景气调整权重 (±3%)
  │
  ├─→ 约束检查模块
  │   ├─ 行业集中度上限 (per sector cap)
  │   ├─ 单个持仓上限 (20%)
  │   └─ Beta 中性性 (target β < 0.1)
  │
  ├─→ 生成建议投资组合
  │   └─ 8 只多头 + 1 只空头
  │
  ├─→ 人工审批 (Dashboard)
  │   ├─ 展示逻辑：当前 vs 建议 (持仓变化、收益预期)
  │   ├─ Victor 可视化审查
  │   └─ 批准/拒绝/手动覆盖 (无超时)
  │
  └─→ 周一开盘交易执行 (Alpaca API)
      ├─ 生成平仓单 (旧持仓)
      └─ 生成建仓单 (新持仓)
```

## 2. Fast Chain — 实时响应链 (Phase 2)

### 2.1 入场流程

```
新闻/价格 Feed (24/7 抓取, 交易时段执行)
  │
  ├─→ G1 快速过滤 (<1s)
  │   ├─ 来源合法性 + 去重 + 实体识别
  │   └─ 95% 淘汰
  │
  ├─→ G2 分类 (<5s)
  │   ├─ FinBERT 情绪: 正面/中性/负面
  │   ├─ 主题分类: 收益预警/拆分/诉讼/并购/政策...
  │   ├─ 紧急度: 低/中/高
  │   └─ 持仓相关度: 0-1
  │
  ├─→ G3 深度阅读 (可选, <30s, ≤10篇/天)
  │   ├─ Claude 全文分析
  │   └─ 输出: 影响评估 ±X%, 建议动作, target/stop 建议
  │
  ├─→ 信号评分
  │   ├─ 情绪 × 重要度 × 紧急度 → 综合分数
  │   └─ 分级 (Tier-1/2/3)
  │
  ├─→ Tier-1 (>85分, 自动执行)
  │   ├─ 可执行: 调仓现有持仓 / 买入新仓 / 紧急平仓
  │   └─ 约束: 单笔 ≤ 5% 总资金, FC 总仓位 ≤ 15%, 仅限 U100
  │
  ├─→ Tier-2 (60-85分, 待审批)
  │   ├─ 推送告警给 Victor, 15 分钟超时
  │   └─ 无响应 → 自动拒绝
  │
  └─→ Tier-3 (<60分, 观察列表)
      └─ 记录但不触发交易
```

### 2.2 FC 入场约束

| 约束维度 | 规则 | 原因 |
|---------|------|------|
| 标的范围 | 仅限 U100 内的股票 | 经过 MC 漏斗验证，基本面和流动性已审核 |
| 单笔上限 | ≤ 5% 总资金 | 限制单次新闻驱动的风险敞口 |
| FC 总仓位 | ≤ 15% 总资金 | FC 是辅助角色，不能喧宾夺主 |
| 日度频率 | Tier-1 ≤ 2 次/天, 冷却期 30 分钟 | 防止过度交易 |
| 资金来源 | 从 MC 组合等比缩减腾出 | 不预留现金，保持 MC 满仓效率 |

### 2.3 FC 退出策略 (两阶段)

**Phase 1 — TTL 自然过期**

FC 的每笔新仓自带 TTL (默认 3 个交易日)。

```
FC 买入 (带 TTL = 3 天)
  │
  ├─→ TTL 期间: 持有, 监控价格
  │     └─ 防守机制仍生效 (3σ 自动平仓)
  │
  ├─→ MC 周度重算发生在 TTL 内:
  │     ├─ MC 选中该股 → 仓位并入 MC, TTL 取消
  │     │                 按 MC 生命周期管理 (持有到下周 MC)
  │     └─ MC 未选中   → TTL 到期自动平仓
  │
  └─→ TTL 到期, MC 未介入 → 自动平仓
```

**Phase 2 — 智能退出 (LLM + ML/RL)**

FC 买入时，系统生成一组退出参数，由三层信号协同决定退出时机。

```
FC 买入时:
  │
  ├─→ LLM 输出初始退出参数
  │     ├─ target_pct: 止盈目标 (如 +5%)
  │     ├─ stop_pct:   止损线 (如 -3%)
  │     ├─ max_hold:   最大持有天数 (如 5 天)
  │     └─ confidence: 预测置信度 (0-1)
  │
  ├─→ ML 模型校准 (LightGBM / XGBoost)
  │     ├─ 输入: 新闻特征 + 股票因子 + 宏观状态
  │     ├─ 输出: 预测未来 N 日收益分布
  │     └─ 修正: 根据收益分布调整 LLM 的 target/stop
  │              (如 LLM 说 +5% 但 ML 预测上行空间仅 +2% → 收紧 target)
  │
  └─→ RL 模型动态管理 (Phase 2 后期)
        ├─ State: 当前盈亏, 持有天数, 市场波动率, 新闻更新
        ├─ Action: 继续持有 / 部分平仓 / 全部平仓 / 调整 stop
        └─ Reward: 实现收益 - 持仓成本 - 机会成本

持仓期间:
  │
  ├─→ 价格触及 target → 止盈平仓
  ├─→ 价格触及 stop   → 止损平仓
  ├─→ 持有天数 = max_hold → 市价平仓
  ├─→ 3σ 防守触发 → 立即平仓 (优先级最高)
  ├─→ 新的反向新闻 (G2/G3 输出反向信号) → 提前平仓
  └─→ MC 周度重算选中 → 并入 MC, 退出参数取消
```

**Phase 2 退出模型训练**

| 组件 | 训练数据 | 输出 | 训练频率 |
|------|---------|------|---------|
| LLM (target/stop 初始值) | 历史新闻事件 + 后续价格走势 | target_pct, stop_pct, max_hold | Prompt Engineering, 无需训练 |
| ML 校准器 (修正 LLM) | FC 历史交易记录 + 实际结果 | 修正后的 target/stop | 每周随 MC 训练一起 |
| RL 退出模型 (动态管理) | FC 持仓期间的状态-动作-收益 | hold/sell/adjust 决策 | 每周离线训练 |

### 2.4 MC 与 FC 交互规则

```
┌─────────────────────────────────────────────────────────────────┐
│                    MC ←→ FC 仓位生命周期                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  场景 1: FC 买入新股, MC 后续选中                                  │
│  ───────────────────────────────────                              │
│  Mon: FC 买入 AAPL 5% (TTL=3d / target=+5%)                     │
│  Fri: MC 重算, AAPL 入选 8L → FC 仓位并入 MC                      │
│       退出参数取消, 按 MC 权重调整, 下周 MC 管理                    │
│                                                                   │
│  场景 2: FC 买入新股, MC 未选中                                    │
│  ───────────────────────────────                                  │
│  Mon: FC 买入 TSLA 5% (TTL=3d / target=+5%, stop=-3%)           │
│  P1: Wed TTL 到期 → 自动平仓                                      │
│  P2: 价格先到 target/stop → 触发平仓 (可能 Tue 就卖了)            │
│                                                                   │
│  场景 3: FC 调仓 MC 现有持仓                                      │
│  ─────────────────────────────                                    │
│  MC 持有 NVDA 12%, FC 因负面新闻减仓至 7%                          │
│  Fri: MC 重算 → MC 重新计算 NVDA 权重, FC 调整自动归位              │
│                                                                   │
│  场景 4: 防守触发 (优先级最高, 覆盖一切)                             │
│  ─────────────────────────────────────                            │
│  任何持仓 (MC 或 FC) 价格 >3σ → 立即平仓, 无需审批                  │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

## 3. 审批工作流

- **周度再平衡 (MC)**：推送通知 → Victor 审查逻辑 (展示收益预期、持仓变化对比) → 批准/拒绝/覆盖 → 无超时限制
- **FC Tier-1 (>85分)**：自动执行，事后推送通知给 Victor
- **FC Tier-2 (60-85分)**：告警 → 15 分钟审批窗口 → 无响应自动拒绝
- **权限设计**：Victor 为唯一审批者，防止越权交易

→ 参考：research-rl-algorithms.md §4（Phase 1 推荐）

## 4. 交易执行策略

- **Broker**：Alpaca API (免佣，API 友好)
- **执行方式**：
  - 周度再平衡：周一开盘市价单
  - 实时信号：立即执行（Tier-1）或待命（Tier-2）
  - 应急防守：立即市价平仓
- **幂等性**：每个交易任务生成唯一 order_id，支持重试而不重复下单
- **融资管制**：
  - 多头：直接持仓
  - 空头：使用反向 ETF 代理 (例 TQQQ/SQQQ) 或融券（如可用）

## 5. 风险控制机制

| 风控措施 | 阈值 | 触发条件 | 动作 |
|---------|------|--------|------|
| 尾部止损 | ATR × 1.5 | 单个持仓下跌超过阈值 | 自动平仓 |
| 应急加速上限 | 2次/天 | Tier-1 信号触发加速 | 超过上限则停止 Tier-1 |
| 加速冷却期 | 30分钟 | 每次 Tier-1 执行后 | 进入冷却期禁止新的 Tier-1 |
| 单个持仓上限 | 20% | 任何时刻 | 强制上限，不可超越 |
| 行业集中度 | 3只/行业 | 任何时刻 | 防止行业过度集中 |
| 日度交易金额 | $50k | 应急措施 | 防止一天内过度交易 |

## 6. 日志与审计

每笔交易记录：
- 执行时间、order_id、持仓变化
- 触发来源 (周度/Tier-1/Tier-2/防守)
- 信号得分、最终权重
- 成交价格、手续费、滑点估计
